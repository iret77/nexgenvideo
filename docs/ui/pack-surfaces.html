Pack-contributed cockpit surfaces

A format pack can give its own measured/structured artifacts a persistent home in the cockpit — data that the generic tabs (Story, Bible, Shotlist, Review) can't hold because they are format-agnostic. The first case is the musicvideo pack's song analysis (tempo map, key, section boundaries, beat grid), which today lives only on disk and scrolls away in chat.

Why (grounded in the current code)

  • The cockpit routes each phase to a declared surface — and knows exactly three: prose→Story, review→Review, choice→in-chat (PipelinePanelView.surfaceIcon). There is no surface type for pack-specific structured data.
  • The analysis phase measures BPM / beats / downbeats / key onto analysis/<song>.json (BeatThisDetector + DSP) and the agent's beat planning consumes it (BeatAssembly) — but nothing renders it. It is invisible (on disk) or ephemeral (chat).
  • Per the platform split (generic Swift host + thin format-packs), a pack must not ship a native view. So a pack surface is declarative: the pack supplies data + a layout descriptor, the host renders it with a fixed vocabulary of cockpit primitives.

Mechanism — declarative pack surfaces

  • The pack's UI contract gains cockpitSurfaces: [{ id, title, symbol, phase, dataFile, layout }]. layout is an ordered list of host primitives bound to fields of dataFile. The host owns the rendering; the pack owns the data + mapping.
  • Host primitive vocabulary (the only things a pack surface may compose): statRow (labelled value tiles), beatTimeline (beats + downbeats + section bands over a duration), sectionList (labelled time bands), keyValue (flat facts). New primitives are host changes, never pack code.
  • A pack surface is read-only ground truth. Measured values (BPM, beats, downbeats, sections) are never editable here — lyrics label sections, they never move the measured boundaries. Editing belongs to the agent flow, not this panel.

Placement rules

  • A pack surface appears as a contextual tab only when it has content — i.e. the analysis data file exists. No active pack (generic project) or a pack whose analysis hasn't run yet → the tab is not rendered at all; it never sits empty. First-run progress is reported in the chat, where the analysis is triggered. Budget: ≤5 generic + ≤1 pack, present only while it has something to show.
  • Position: immediately after Pipeline, before the generic artifact tabs — Pipeline · Analysis · Story · Bible · Shotlist · Review. Rationale: it is upstream ground truth the whole production reads from, so it sits next to the control room, not amid the downstream artifacts.
  • The pack tab is visually marked as pack-scoped (a PACK dot in the accent colour --pack) so it never reads as a first-party generic tab.
  • It is also reachable from Pipeline: the analysis phase row's route button opens this surface (reusing the existing surfaceIcon routing), so approving the phase and inspecting its result are one gesture.

States that must be covered

  • Not shown — no analysis data yet: generic project, or pack active but analysis not run. The tab does not exist; there is never an empty Analysis tab. (First-run progress lives in the chat.)
  • Loaded — data present: stats + beat timeline + section list + provenance.
  • Degraded — data present but no reliable beat grid: show what was found (duration, key) and name the limitation; never present an empty grid as truth.
  • Re-measuring — data present and a fresh analysis is running: in-panel progress over the last-known values; the tab does not vanish mid-run.

Acceptance criteria

  • The Analysis tab appears only once analysis has produced data. Before that (analysis not run) and for generic projects the bar is the 5-tab generic bar — never an empty Analysis tab. When data exists the bar is Pipeline · Analysis · Story · Bible · Shotlist · Review.
  • The Analysis tab and the Pipeline analysis-row route open the same panel.
  • The panel is rendered entirely from host primitives fed by analysis/<song>.json; no pack-supplied view code is loaded.
  • No control in the panel mutates BPM / beats / downbeats / sections (read-only ground truth).
  • Every state above renders without layout break at 1280px and 1040px cockpit widths.

Flow — how a pack surface reaches the screen

flowchart LR
  A[Pack UI contract\ncockpitSurfaces[]] --> B[Host reads contract\non project open]
  B --> C{active pack declares a surface\n& its data file exists?}
  C -- no --> D[no Analysis tab\n(generic, or not run yet)]
  C -- yes --> E[insert contextual\nAnalysis tab after Pipeline]
  F[analysis/<song>.json] --> G[Host primitives\nstatRow · beatTimeline · sectionList]
  E --> G
  H[Pipeline analysis row\nroute button] --> G
  G --> I[Analysis panel\nread-only]
